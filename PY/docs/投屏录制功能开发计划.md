# 投屏录制功能开发计划

> 分支：`feature/screen-recording`
> 日期：2026-06-27
> 目标：在实时投屏窗口中，开启录制后用户的所有点击/滑动操作即被录制为工作流步骤（含操作间等待），类似按键精灵的"录制宏"。停止后步骤直接追加到当前工作流。

---

## 一、背景与现状（代码调研结论）

| 现有模块 | 状态 | 与本需求的关系 |
|---|---|---|
| `PY/core/recorder.py` | **死模块**，全项目无人引用 | 基于 `adb shell getevent` 轮询设备级事件，只解析 tap（无 swipe），不向设备回注输入。与需求方向不符，需重构。 |
| `PY/core/screen_capture.py`（`ScrcpyCapture`） | 投屏核心 | scrcpy 以 `control=false` 启动，**只显示、不能反向控制设备**；有 screencap 回退。解码链路 `scrcpy→ffmpeg(h264→mjpeg)→cv2.imdecode` 有两次转换。 |
| `PY/ui/components/screenshot_picker.py`（`_ImageLabel`） | 静态截图选点器 | 唯一带"点击→坐标"能力的组件，已内置完整坐标映射链（`display→img→device`，基于 `base_resolution`，默认 2400×1080）、缩放、校准。但**不是实时镜像窗口**。 |
| `PY/ui/main_window.py` | 主窗口 | 未把 `frame_captured` 接到任何实时显示控件——**目前不存在可交互的实时投屏窗口**。 |
| `PY/core/adb_core.py` | ADB 注入 | 已有 `tap/swipe/keyevent/input_text`，但走 `adb shell input`，每次 fork 进程延迟 ~80–150ms，**不可接受**。 |
| `PY/core/step_executor.py` + `workflow_panel.py` | 步骤格式与执行 | 步骤格式已确认：`tap={type,x,y,comment,wait_after}`、`swipe={type,x1,y1,x2,y2,duration,comment}`，另有 `tap_point/wait/keyevent`。 |

## 二、需求与决策

| # | 决策点 | 最终决策 |
|---|---|---|
| 1 | "点击等待"语义 | **操作即被录制**，无需显式"插入等待"按钮；操作间等待时长自动记录 |
| 2 | 录制时是否实时注入设备 | **边操作边注入**（设备实时响应），同时记录为步骤 |
| 3 | 延迟要求 | **不可接受 `adb input` 延迟**，需低延迟方案（见第三节） |
| 4 | 录制结果去向 | **直接追加到当前工作流** |

---

## 三、延迟优化方案（不降画质）

当前瓶颈在两端：**注入**用 `adb shell input`（每次 fork 进程 ~80–150ms）；**解码**链路有两次转换。分别优化。

### A. 注入通道：启用 scrcpy 控制协议（核心，延迟 ↓ ~90%）

- `screen_capture.py` 启动参数 `control=false` → `control=true`。
- 通过**同一 TCP socket** 写控制消息（scrcpy 2.0 协议，二进制）：`type=0x02 INJECT_TOUCH_EVENT`，含 action（DOWN/MOVE/UP）、pointer_id、x、y、pressure、action_button。
- 实现 `ScrcpyControl`：
  - `inject_tap(x,y)` = DOWN + UP 两条消息。
  - `inject_swipe(x1,y1,x2,y2,duration)` = DOWN + 多条 MOVE（按 duration 插值）+ UP。
- 延迟 ~5–15ms（同 socket、无进程 fork）。
- **回退**：control 协议失败/不支持时降级 `adb input`，保证可用性。

### B. 显示通道：解码链路精简（延迟 ↓ ~10–25ms）

- ffmpeg 输出从 `mjpeg` 改为 **`rawvideo (bgr0)`**：省掉 mjpeg 编码 + JPEG 解码两步，直接 `np.frombuffer` 成帧，CPU 减半。
- 保持画质：
  - `max_size=1080` 不变。
  - `bit_rate` 由 2Mbps 提到 **6Mbps**（1080p 高画质）。
  - 新增 `i_frame_interval=10`（关键帧更密，加快重连首帧，带宽略增可接受）。
- 进阶可选（v2，本期不做）：PyAV + 硬解（nvdec/qsv），依赖重。

### C. 帧管线：覆盖式缓冲 + 节流拉取

- 解码线程：**覆盖式**写 `_current_frame`（不排队，丢中间帧），消除积压。
- UI 线程：`QTimer@30fps` 主动拉取最新帧渲染，不靠 `frame_captured` 信号逐帧推送。
- 去掉冗余 `.copy()`：解码写一份，UI 拉取时 copy 一次。

### 预期效果

注入 ~10ms（原 ~100ms）、显示 ~15ms（原 ~35ms）、端到端 ~25–40ms，可流畅交互录制。

## 四、架构设计

```
[ScrcpyCapture.frame_captured] ──帧──▶ MirrorWidget(实时显示, QTimer@30fps 拉取)
                                          │ 鼠标 press/move/release
                                          ▼
                                   坐标映射(display→device)
                                          │
                            ┌─────────────┴─────────────┐
                  录制关闭: 仅注入                   录制开启: 注入 + 记录为步骤
                            │                             │
                   ScrcpyControl.inject            StepRecorder(去抖/分类 tap|swipe|wait)
                                                          ▼
                                          停止 → 生成 steps → 直接追加到当前工作流
```

**关键决策**：
- 注入走 scrcpy 控制协议（低延迟），失败降级 `adb input`。
- 新建 `MirrorWidget` 而非改造 `ScreenshotPicker`（语义不同：实时交互 vs 静态选点），但抽公共坐标映射避免重复。
- `recorder.py` 由 getevent 轮询重构为窗口事件驱动。
- 等待时长写入上一步 `wait_after`，复用现有字段，回放自然衔接。
- 坐标用设备分辨率，跨设备回放靠 `base_resolution` 重映射。

## 五、任务分解（分阶段）

### 阶段 1：实时镜像控件 `MirrorWidget`

- 新建 `PY/ui/components/mirror_widget.py`，从 `_ImageLabel` 抽离坐标映射（`_display_to_img`/`_img_to_device`/`base_resolution`/缩放）复用。
- QTimer@30fps 拉 `ScrcpyCapture` 最新帧渲染。
- 捕获 `mousePressEvent/mouseMoveEvent/mouseReleaseEvent`，发出结构化交互信号（按下点、轨迹、抬起点 + 时间戳）。

### 阶段 2：`ScrcpyCapture` 升级（延迟优化落地）

- 启动参数改 `control=true`，新增 `bit_rate=6000000`、`i_frame_interval=10`。
- 新增 `ScrcpyControl`：实现 `inject_tap/inject_swipe`（scrcpy 控制协议二进制消息），失败降级 `adb input`。
- ffmpeg 输出改 `rawvideo`，解码线程改覆盖式缓冲。
- 新增 `inject_tap/inject_swipe` 公开方法供 `MirrorWidget` 调用。

### 阶段 3：`StepRecorder`（重构 `recorder.py`）

- 由 getevent 轮询改为**窗口事件驱动**：接收 `MirrorWidget` 交互信号。
- 手势分类：
  - 按下点 ≈ 抬起点（位移 < 阈值）→ `tap`。
  - 有位移 → `swipe`（含 duration）。
- 操作间等待：相邻操作时间差 → 写入上一步 `wait_after`。
- 去抖过滤（忽略极短抖动/误触）。
- `start/stop/is_recording` + `event_recorded(dict)` 信号，`stop` 返回 `List[dict]`。

### 阶段 4：UI 接入与结果落地

- `main_window.py`：实例化 `MirrorWidget`，接 `frame_captured`；新增"投屏录制"入口（工具栏按钮/Tab）。
- 录制开关：开始/停止 + 状态指示（红点+计时）；录制中注入走 `ScrcpyControl`，事件同时送 `StepRecorder`。
- 录制中禁用无关编辑防冲突。
- 停止 → `StepRecorder.stop()` 返回 steps → 直接追加到当前工作流（`workflow_panel`/`config_manager`），`comment` 自动标"录制 #n"。

### 阶段 5：测试与验收

- 单测：`StepRecorder` 手势分类/去抖/等待、坐标映射、`ScrcpyControl` 报文构造。
- 集成测：`tests/test_integration.py` 加录制→回放用例。
- 手动验收：`PY/docs/cast-acceptance.md` 补录制验收项。

## 六、风险与注意事项

| # | 风险/注意点 | 应对 |
|---|---|---|
| R1 | scrcpy 控制协议二进制打包必须精确（字节序、字段长度），错位会断连 | 加字段校验 + 失败降级 `adb input`；单测覆盖报文构造 |
| R2 | `control=true` 改动影响现有投屏链路，需保证回退路径 | 降级机制完备；改动集中在 `ScrcpyCapture` 内部 |
| R3 | `rawvideo` 改动要同步改解码线程的解析逻辑（去掉 JPEG 边界扫描） | 解码线程重写为按帧大小定长读取 |
| R4 | 录制坐标必须用设备分辨率，否则跨设备回放错位 | 全程走 `base_resolution` 映射；单测覆盖 |
| R5 | 操作间等待写入上一步 `wait_after`，**最后一步的等待**无下文承接，需决定保留或丢弃 | 默认保留（回放时尾部等待无害） |
| R6 | 录制中需禁用工作流编辑，防止步骤列表并发修改冲突 | 录制开始时锁编辑，停止时解锁 |
| R7 | 手势分类阈值（tap vs swipe 的位移判定）需可调，不同设备触控精度不同 | 阈值做成常量，按 `base_resolution` 归一化 |
| R8 | scrcpy 协议版本差异（2.0 字段布局），不同 server 版本可能不兼容 | 锁定 `lib/scrcpy-server.jar` 版本，启动时校验 |

## 七、验收标准

1. 开启投屏录制后，在投屏窗口的点击/滑动实时反映到设备，且被记录为 `tap`/`swipe` 步骤。
2. 操作间等待自动记录为上一步的 `wait_after`。
3. 停止录制后，步骤直接追加到当前工作流，可在工作流编辑器中查看、编辑、回放。
4. 录制的步骤在**同分辨率设备**回放，操作序列与录制时一致。
5. 端到端延迟 ≤ 40ms（注入 + 显示）。
6. scrcpy 控制协议失败时自动降级 `adb input`，功能不中断（仅延迟升高）。
7. 单测、集成测全部通过。
